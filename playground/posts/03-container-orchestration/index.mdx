# 容器编排：从 Docker 到 Kubernetes

容器化不是目的，而是手段。真正的目标是让应用的交付和运维变得可预测、可重复。

## 为什么选择容器化？

传统部署面临的核心问题：

- **环境不一致** — "在我机器上能跑" 是最常见的运维噩梦
- **资源利用率低** — 一台物理机跑一个应用，CPU 利用率常年不到 10%
- **扩缩容慢** — 新增一台服务器需要几小时甚至几天

## Docker 镜像优化

镜像大小直接影响部署速度和存储成本。

### 多阶段构建

```dockerfile
# Stage 1: Build
FROM node:20-alpine AS builder
WORKDIR /app
COPY package.json pnpm-lock.yaml ./
RUN corepack enable && pnpm install --frozen-lockfile
COPY . .
RUN pnpm build

# Stage 2: Runtime
FROM node:20-alpine AS runtime
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
EXPOSE 3000
CMD ["node", "dist/index.js"]
```

### 镜像层缓存策略

把变化频率低的层放在前面，变化频率高的放在后面。这样 Docker 可以最大化利用缓存。

## Kubernetes 集群设计

空岛云的 K8s 集群采用三层架构。

### 节点规划

| 节点类型 | 数量 | 规格 | 用途 |
|---------|------|------|------|
| Master | 3 | 4C8G | 控制面 |
| Worker | 5-20 | 8C16G | 业务负载 |
| Infra | 3 | 4C8G | 监控/日志 |

## 服务网格

Istio 为我们提供了统一的流量管理、安全通信和可观测性。

## 总结

容器编排的核心价值在于标准化和自动化。
